# 分支管理策略

```
  实际开发中，我们因该按照几个基本原则进行分支管理：
  首先master分支应该是非常稳定的，也就是说master分支仅用来发线上版本（release），平时开发不应该在master分支上改动；
  所有的功能开发都在dev分支上，也就是说dev分支是不稳定的，它先于master分支，等到功能开发完并测试结束后再把dev分支合回到master上重新打release tag发版
  所有的开发人员从dev分支上拉取自己的开发分支，每个人在自己的分支上开发新功能，开发完并测试完成后合到dev
  其中，dev分支对应的是集成环境
        master分支对应的是线上环境
        自己的分支对应的是功能测试环境
```

# 如何修复线上紧急bug？

```
  应该在最新的【记得pull一下】master分支上拉取分支，命名为hotfix-年月日或修复的bug名称（可以是对应在redmine里面的bug号码），
  然后开发完了以后，把master分支合到hotfix分支并打test tag，测试结束后再把hotfix合回master并在master上打release tag，
  最后让测试在线上验证bug，通过没问题，没通过则撤回revert重新检查
```

# 如何开发新功能？

```
  对应的开发应该在最新的【记得pull一下】dev分支上拉取新的分支，并且命名feature-年月日或功能名称，然后在feature上开发；
  开发完了以后在自己的feature上打上dev tag 发到内网环境进行功能测试；
  测试通过后需要把最新的【记得pull一下】dev分支合并到自己的feature分支上【这样做的目的是如果一旦发现冲突，在自己的分支上处理，
  避免在dev分支上处理，不然会带来很麻烦的事情】，最后再把自己的feature分支合回dev分支【这个时候肯定没有冲突，
  因为在从dev合到feature的时候有冲突已经处理好了】，打上test tag 发布到集成环境 进行集成测试
```

# 如何合并代码？

```
  一般操作都是把自己的开发分支合并到主线，这时候应该切换到主线执行pull操作，然后再把最新的主线合并到自己的开发分支，
  有冲突解决冲突，没有冲突更好，完了以后再把自己的分支合到主线；把冲突放到自己的分支去处理，避免直接在主线上处理冲突；
  处理不当可能会导致线上环境有问题；
```

# 如何管理自己的分支？

```
  因为自己的开发分支是从dev上拉取的，而dev上每天都有不同的人往上面推送新功能，所以，每天第一件事应该是把dev上的更新合并到自己
  的开发分支上，避免积少成多上线的时候解决大量冲突；
```

# master分支谁来管理？

```
  master分支因该统一由指定的一个人或几个人管理，因为master不是开发分支，所以一般只有发布上线的时候才会在master上打release tag；
  master应该是跟线上环境始终保持统一的
```

# 注意

```
  开发过程中，切记多commit少push，有时候开发一个小功能push量非常多，如果出现什么问题回滚都不好找节点；
  一个小功能开发一个push就够了
```
